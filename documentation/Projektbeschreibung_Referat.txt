## 1. Einleitung und Projektvision

Dieses Dokument beschreibt die Architektur, den Entwicklungsprozess sowie die programmiertechnische Umsetzung des Backends für ein kombiniertes Tippspiel und Bingo zur Fußball-Weltmeisterschaft 2026, entwickelt mit C# und dem .NET-Framework.

### 1.1 Das reale Projekt
Das Projekt ist nicht lediglich ein theoretisches Konstrukt für die Prüfungsleistung, sondern zielt auf einen produktiven Einsatz ab. Die Anwendung ist für eine Teilnehmerzahl von ca. 200 bis 400 Tippern konzipiert. In einer solchen Größenordnung, in der zudem mit echten finanziellen Einsätzen (einem Gesamtpool von ca. 1.800 €) gespielt wird, sind Zuverlässigkeit, Fehlerfreiheit und Nachvollziehbarkeit der Punkte- und Gewinnverteilung zwingende Grundvoraussetzungen.

Das System kombiniert das klassische Vorhersagen von Spielergebnissen (mit Punkten für Exaktes Ergebnis, Tordifferenz, Tendenz) mit einer innovativen Bingo-Komponente, in der Ereignisse während der Spiele (z.B. "Rote Karte in Gruppe A") eine individuelle 5x5 Bingo-Karte füllen. Diese Kombination führt zu einem hochkomplexen Regelwerk, das durch eine C# *Calculation Engine* täglich verlässlich abgewickelt werden muss.

### 1.2 Begründung der Technologie-Entscheidung (Abweichung von BANF1)
Die Vorlesungsmaterialien ("Einheit 2: Windows-Forms-Anwendung und WPF") fokussieren unter anderem auf die Erstellung von Desktop-Anwendungen mit grafischen Benutzeroberflächen wie WPF. 

**Warum keine WPF-Applikation?**
Die Zielgruppe dieses Projekts (bis zu 400 Personen) wird vorrangig über Smartphones, Tablets sowie teilweise Mac-Rechner interagieren. Eine klassische Windows Desktop-Anwendung (WPF) kann diesen Use-Case nicht bedienen, da es den Nutzern nicht zugemutet werden kann, eine Software herunterzuladen und zu installieren, insbesondere da es sich um mobile Nutzer handelt. 

Daher ist die Entscheidung auf ein reines **C# Backend (ASP.NET Core Minimal API)** gefallen, das als Web-Anwendung agiert. Die Client-Schicht (Frontend) ist in HTML und JavaScript realisiert. C# wird somit als reine Server-Technologie für die komplexe Business-Logik eingesetzt – ein Szenario, das in der modernen C#-Programmierung den Standard darstellt. C# behält dabei exakt dieselben objektorientierten Eigenschaften, die in den Kursen eingefordert werden (SOLID, Vererbung, Collections), jedoch zugeschnitten auf eine realistisch nutzbare Architektur.

## 2. Entwicklungsprozess und Anforderungsanalyse

Der Entwicklungsprozess orientierte sich iterativ an den Erfordernissen eines echten IT-Projekts.

### 2.1 Requirements Engineering
Bereits vor der ersten Code-Zeile mussten die Domänenregeln niedergeschrieben werden. Die Herausforderungen bestanden darin:
- Genaue Zuweisung von Punkten (Bingo: Reihen, Felder / Klassisch: Matchpunkte).
- Handhabung der komplexen Geldtöpfe (Zwischengewinne für Gruppencluster, Spezialtöpfe für Bingo, Endgewinne).
- Auflösen von Edge-Cases (z.B. Gleichstand bei Punkteauswertung -> Tie-Breaker Regeln).

### 2.2 Systemmodellierung und UML
Analog zur Kurseinheit über Systementwicklung und Modellierung wurden die fachlichen Einheiten in klar abgegrenzte Klassen überführt. Die Modellierung trennt streng zwischen:
1. **Domänen-Objekten:** Datencontainer wie `user`, `TournamentData`, `BingoCard` (ohne Eigenlogik).
2. **Business-Logik (Dienste):** Berechnungs- und Bewertungsklassen (`Evaluator`, `Distributor`).
Diese Entkopplung bildet den fundamentalen Schritt auf dem Weg zu lose gekoppelten C#-Systemen.

## 3. Architektur und Klassendesign (BANF4)

Das Projekt nutzt weitreichend die in den Vorlesungen besprochenen objektorientierten Prinzipien. Um der Komplexität des Systems Herr zu werden, ist das C#-Projekt in fünf stark separierte Schichten (Namespaces) nach dem "Clean Architecture" Prinzip gegliedert.

### 3.1 Die 5-Schichten-Architektur

1. **Domain (`TippspielApp.Domain`)**: 
   Enthält ausschließlich Klassen, die Daten repräsentieren (DTOs = Data Transfer Objects). Hier gibt es keine Berechnungen. Wichtige Klassen sind hier `User`, `TournamentData` und `BingoCard`.
2. **Scoring (`TippspielApp.Scoring`)**: 
   In diesem Namespace finden sich alle Klassen, die Punkte auf Basis der Spieldaten vergeben, wie der `ClassicEvaluator` (Spielergebnisse) und der `BingoEvaluator`.
3. **Finance (`TippspielApp.Finance`)**: 
   Sämtliche Logik zur Preisgeldverteilung (PotMath, MainPotDistributor). Da ein Teilnehmer nur Geld gewinnt, nachdem die Punkte ausgewertet wurden, ist die fachliche Trennung essenziell. 
4. **Application (`TippspielApp.Application`)**: 
   Hier befindet sich der `RankingCalculator`. Er agiert als "Director", der die Klassen der restlichen Schichten orchestriert. Er lädt die Daten, triggert nacheinander die verschiedenen Evaluatoren, sortiert, feuert die Finance-Komponente ab und erzeugt schlussendlich die Ranglisten.
5. **Infrastructure (`TippspielApp.Infrastructure`)**: 
   Übernimmt reine I/O Aufgaben, sprich das Laden oder Speichern von Dateien in das Dateisystem. Hier liegt die `JsonDataStore` Klasse.

## 4. Objektorientierung und SOLID in C# (BANF2)

Die strikte Anwendung der objektorientierten Theorie und der SOLID-Kriterien bildete die Grundlage der Entwicklung.

### 4.1 S - Single Responsibility Principle (SRP)
Jede Klasse in diesem System hat exakt eine Verantwortung. Ein Negativbeispiel wäre eine Klasse `User`, die nicht nur ihre Daten hält, sondern auch eine Methode `.CalculateMyPoints()` besitzt und im schlimmsten Fall sich selbst speichern könnte `.SaveToFile()`. 
Im vorliegenden Projekt ist dies stattdessen verteilt:
- `User`: Speichert nur die Eigenschaft (z.B. seinen Namen und die getippten Resultate).
- `ClassicEvaluator`: Weiß lediglich, wie man Fußallergebnisse auswertet und Punkte der `User` ändert.
- `JsonDataStore`: Weiß nur, wie man Objekte in das Dateisystem ablegt.

### 4.2 O - Open/Closed Principle (OCP)
Systeme sollen "offen für Erweiterungen, geschlossen für Modifikationen" sein. Dies wird in C# primär durch Interfaces gelöst. Der `RankingCalculator` muss über alle Punktevergabesysteme iterieren. 
Statt im Calculator den `ClassicEvaluator` und den `BingoEvaluator` einzeln aufzurufen, verwalten wir alle Evaluatoren als Liste des Interfaces `IEvaluator`. 
Sollte die WM 2026 eine neue Regel fordern (z.B. "Bonus-Punkte für gelbe Karten"), muss die Klasse `RankingCalculator` nicht verändert werden. Es wird lediglich eine neue Klasse `YellowCardEvaluator : IEvaluator` entwickelt und der Evaluatoren-Liste hinzugefügt. Das System ist somit erweiterbar, ohne Kernschichten anpassen zu müssen.

### 4.3 L - Liskov Substitution Principle (LSP) & Vererbung
In der *Scoring* und *Finance* Schicht verwenden wir Basisklassen (`EvaluatorBase`, `PrizeDistributorBase`). Alle konkreten Klassen wie `ClassicEvaluator` oder `SpecialBetEvaluator` leiten davon ab. Das Vorlesungskapitel über Polymorphie findet hier konsequente Anwendung: Ein Programm kann jederzeit eine Referenz der Basisklasse oder des Interfaces für Berechnungen nutzen, vertrauend darauf, dass jede abgeleitete Unterklasse sich spezifikationsgerecht verhält.

## 5. Collections und LINQ (Kurseinheit 4 Bezug)

Die Vorlesungsmaterialien behandeln unter *Datenstrukturen / Collections* die Vor- und Nachteile von Arrays, Listen usw.

### 5.1 System.Collections.Generic - Typisierte Listen & Hash-Mengen
Die C#-Standardbibliothek stellt starke Generics zur Verfügung.
Anstatt klassischer Arrays verwenden wir im System durchwegs `List<T>`. 
Ein interessanter Anwendungsfall der C# Collections im Projekt ist die Optimierung im `BingoEvaluator`. Die Turnierdaten liefern teilweise umfangreiche Listen an Ereignissen ("OccurredBingoEvents"). Um für die 25 Zellen jedes Users zu prüfen, ob dieses spezfische Event eingetreten ist, nutzen wir keine verschachtelten For-Schleifen. 
Die C# Struktur `HashSet<string>` wird hier initialisiert. Da diese intern Hash-Maps verwendet, läuft eine Anfrage `eventSet.Contains(cell.EventId)` in O(1) statt in O(n) Zeit. Dies ist im Hinblick auf den "Big-O"-Komplexitäts-Begriff und 400 Teilnehmer ein immenser Rechenvorteil.

### 5.2 LINQ (Language Integrated Query)
LINQ ist eines der stärksten Kern-Features in C# und wurde ausgiebig für Aggregation, Transformation und Filterung genutzt. Dies verhindert unübersichtlichen Quellcode durch Reduktion von for-Schleifen.
*Beispiel:* Im Sortieralgorithmus des `RankingCalculator` werden Punktegleichstände aufgelöst. 
Mit LINQ-Queries wie `card.Cells.Where(c => c.IsFulfilled).Select(c => c.Position)` werden fließend und elegant Teildaten extrahiert, um z.B. nur die Positionen der bereits vom Nutzer richtig getippten Bingofelder zurückzugeben. Auch Aggregationen wie `.Sum()` zur Summenberechnung von totalen Punkten werden dank LINQ-Erweiterungsmethoden genutzt.

## 6. Persistenz durch JSON (BANF3)

Eine Anforderung an die Applikation ist das Abspeichern des Programmzustandes. Die klassische Vorlesungen deuten auf manuelle Textdatenspeicherung oder XML hin. In diesem Projekt wurde jedoch mit `System.Text.Json` gearbeitet, der modernsten Form der Daten-Persistenz im Web-Umfeld.

Anstatt hartcodierter Daten oder temporärer Speichern im RAM greift das System auf JSON zu. Alle User, Tipps und Turnierergebnisse liegen in Dateien wie `users.json` und `tournament_data.json`.
Die Klasse `JsonDataStore` (Schicht: Infrastructure) deserialisiert den JSON-Text über `typeof(User)` vollständig in C#-Objekte. 
Dies hat enorme Vorteile:
1. **Zustandssicherheit:** Auch wenn der Webserver bei der Berechnung der Punkte abstürzt, sind die Ur-Tipps in Form der JSON sicher.
2. **Einfache Wartbarkeit:** Zur Korrektur kann im Notfall direkt in Klartext die JSON eingesehen werden.
3. Die Serialisierungs-Optionen (`PropertyNameCaseInsensitive = true`) kompensieren minimale Schreibfehler zwischen C#-Code (CamelCase / PascalCase) und dem JSON-Objekt.
Nach jeder Berechnung schreibt C# ein komplettes Abbild (Rankings, Gewinne) wieder als `ranking_current.json` nativ heraus.

## 7. Ausblick und zukünftige Meilensteine

Das System ist in seinem Kern als voll funktionsfähige *Auswertungsmaschine* entwickelt und belegt tiefgreifendes Verständnis der C# Kernprinzipien. Für das finale Release des Live-Projektes sind jedoch folgende Schritte zeitnah geplant:

1. **Cloud-Deployment auf Azure:**
   Das Backend wird über eine CI/CD-Pipeline als fully-managed "Azure Web App" deployed. Dadurch ist gewähreistet, dass Hunderte Zugriffe gleichzeitig skalierbar abgefangen werden können.
2. **Ersetzung der JSON-Dateien durch Azure SQL (Entity Framework Core):**
   Auf Basis der Clean Architecture Struktur muss für den Wechsel auf eine Datenbank keinerlei Berechnungs-Logik verändert werden. Das `JsonDataStore` wird einfach durch ein `DatabaseDataStore` ausgetauscht, das via Entity Framework Core C#-Objekte direkt auf tabellarische SQL-Datenbanken und Relationen mapped. Die JSON Persistenz fungiert momentan als extrem valides MVP (Minimum Viable Product).
3. **Barrierefreiheit / Funktion für "Ältere Mitspieler":**
   Nicht jeder User dieser Tipp-Gemeinschaft ist web-affin. Es ist eine Funktion vorgesehen, über PDF-Generierungs-Bibliotheken (in C# abgebildet), das Bingo-Feld aus dem HTML dynamisch in druckbare PDFs umzuwandeln.
4. **Dashboard-Frontend (React/Vue oder Vanilla JS/HTML):**
   Damit Benutzer nachverfolgen können *warum* sie bestimmte Punkte erhalten haben, wird derzeit an einer interaktiven Benutzeroberfläche gebaut. Diese UI konsumiert über REST-API (HTTP) letztlich ausschließlich die von unserer C#-Engine generierten Daten und JSON-Antworten.
